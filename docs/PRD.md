# PRD: 근무시간 관리 시스템

## 1. 개요

근무시간 관리 시스템은 사용자가 본인의 근무시간을 입력하고 조회하며, 지정한 기간의 근무 내역을 정리할 수 있는 웹 애플리케이션입니다. 외부 시스템에서 허용된 IP와 유효 기간 안에 웹훅을 호출하면, 서버는 지정된 근무기간을 요약해 텔레그램 메시지로 전송합니다.

근무 기록은 임금, 계약, 증빙과 연결될 수 있으므로 변경 이력, 접속 기록, 감사 로그를 남겨 데이터 정합성과 보안을 보장해야 합니다.

## 2. 목표

- 사용자가 회원가입 후 본인의 근무시간을 입력하고 조회할 수 있어야 합니다.
- 지정한 기간의 근무 기록을 합산, 정리, 요약할 수 있어야 합니다.
- 지정된 기간에 허용된 특정 IP에서 웹훅을 받으면 텔레그램으로 근무기간 요약 메시지를 전송해야 합니다.
- 데이터가 타인에 의해 변조되지 않도록 권한, 감사 로그, 변경 이력, 무결성 검증을 보장해야 합니다.

## 3. 사용자

| 사용자 | 설명 |
| --- | --- |
| 일반 사용자 | 본인의 근무시간을 입력, 조회, 정정하는 사용자 |
| 관리자 | 사용자별 근무 기록, 감사 로그, 웹훅 처리 내역을 확인하는 사용자 |
| 외부 시스템 | 정해진 IP와 기간 안에서 근무 요약 웹훅을 호출하는 시스템 |

## 4. 핵심 기능

### 4.1 회원가입 및 인증

- 이메일 또는 아이디 기반 회원가입
- 비밀번호 해시 저장
- 로그인, 로그아웃
- 사용자별 근무 기록 분리
- 관리자 계정은 전체 사용자 기록 조회 및 감사 로그 확인 가능

초기 정책:

- 인증 방식은 PHP session 기반으로 시작합니다.
- 세션 쿠키는 HTTP-only로 설정합니다.
- 운영 환경에서는 HTTPS와 secure cookie를 사용합니다.
- 로그인 성공, 실패, 로그아웃은 감사 로그에 기록합니다.

### 4.2 근무시간 입력

사용자는 다음 방식으로 근무시간을 등록할 수 있습니다.

- 날짜, 시작 시간, 종료 시간 직접 입력
- 휴게 시간 입력
- 메모 입력
- 자유 형식 텍스트를 AI로 파싱한 뒤 사용자가 확인 후 저장

필수 검증:

- 종료 시간은 시작 시간보다 늦어야 합니다.
- 같은 사용자의 동일 시간대 중복 등록을 방지해야 합니다.
- 야간 근무처럼 날짜를 넘기는 기록은 명시적으로 처리해야 합니다.
- 저장 전 사용자가 최종 확인한 값만 공식 근무 기록으로 반영해야 합니다.

### 4.3 근무시간 조회

사용자는 본인 근무 기록을 조회할 수 있습니다.

- 일별 조회
- 주별 조회
- 월별 조회
- 직접 지정한 기간 조회
- 총 근무 시간, 총 휴게 시간, 실제 근무 시간 합산
- CSV 또는 JSON 다운로드

관리자는 권한에 따라 사용자별, 기간별 조회가 가능해야 합니다.

### 4.4 근무 기록 수정 및 삭제

- 저장된 근무 기록은 임의로 덮어쓰지 않습니다.
- 수정 시 기존 값과 변경 값을 모두 감사 로그에 저장합니다.
- 수정 시 `version`을 1씩 증가시켜 변경 횟수를 추적합니다.
- 삭제는 물리 삭제하지 않고 soft delete로 처리합니다.
- 관리자는 삭제된 기록도 감사 목적으로 조회할 수 있어야 합니다.

### 4.5 웹훅 기반 텔레그램 알림

외부 시스템이 지정한 기간에 허용된 IP에서 웹훅을 호출하면, 서버는 지정된 근무기간의 기록을 정리해 텔레그램 메시지로 전송합니다.

웹훅 요구사항:

- 허용 IP 목록에 포함된 요청만 처리
- 지정된 유효 기간 안의 요청만 처리
- shared secret 검증
- 동일 `requestId` 중복 처리 방지
- 성공, 실패, 거부 내역을 모두 로그로 기록

예시 요청:

```json
{
  "userId": "user_123",
  "from": "2026-05-01",
  "to": "2026-05-31",
  "telegramChatId": "123456789",
  "requestId": "monthly-summary-2026-05-user-123"
}
```

예시 텔레그램 메시지:

```text
[근무시간 요약]
대상 기간: 2026-05-01 ~ 2026-05-31
총 근무일: 18일
총 근무시간: 142시간 30분
총 휴게시간: 12시간 00분
실제 근무시간: 130시간 30분

최근 변경 기록: 2건
마지막 정리 시각: 2026-05-18 09:00
```

### 4.6 보안 및 데이터 정합성

필수 원칙:

- 모든 데이터는 사용자 권한 범위 안에서만 접근 가능
- 생성, 수정, 삭제는 모두 감사 로그에 기록
- 기존 근무 기록을 물리 삭제하지 않고 상태 변경 또는 이력 테이블로 관리
- 변경 전 값과 변경 후 값을 모두 보관
- 로그는 일반 사용자가 수정할 수 없어야 함
- 중요한 요청은 요청 IP, user-agent, 요청 ID, 처리 결과를 함께 기록

권장 무결성 방식:

- 근무 기록 변경 시 `version` 증가
- 감사 로그에 이전 해시와 현재 해시를 저장해 체인 형태로 연결
- 관리자도 감사 로그를 직접 수정하지 못하도록 별도 권한 또는 append-only 저장소 사용
- 데이터베이스 트랜잭션으로 근무 기록과 감사 로그를 함께 저장

## 5. 디자인 시스템

PHP 자체는 UI 디자인 시스템을 제공하지 않습니다. 이 프로젝트는 서버 렌더링 PHP 화면에서 바로 적용하기 쉽고, 폼과 테이블 중심의 업무 화면에 적합한 Bootstrap 5 계열을 기본 디자인 시스템으로 사용합니다.

### 5.1 선택 기준

- PHP 서버 렌더링 페이지에서 바로 사용하기 쉽습니다.
- 폼, 버튼, 테이블, 알림, 모달, 탭, 네비게이션 등 업무 화면에 필요한 기본 컴포넌트가 충분합니다.
- 반응형 grid와 utility class를 제공해 모바일, 태블릿, 데스크톱 대응이 쉽습니다.
- 별도 빌드 도구 없이 CDN 또는 정적 CSS/JS 파일로 시작할 수 있습니다.
- 관리자 페이지와 데이터 관리 화면처럼 기능 중심 UI에 적합합니다.

### 5.2 UI 원칙

- 첫 화면은 로그인 또는 근무시간 대시보드로 시작합니다.
- 사용자의 주요 작업은 근무시간 입력, 기간 조회, 수정 이력 확인입니다.
- 화면은 장식보다 데이터 입력과 확인에 집중합니다.
- 테이블은 날짜, 시작, 종료, 휴게, 실제 근무시간, 상태, 수정 이력을 한눈에 볼 수 있어야 합니다.
- 위험한 작업은 삭제 대신 비활성화 또는 정정 처리로 표현합니다.
- 성공, 실패, 경고, 보안 거부 상태는 색상과 문구를 일관되게 사용합니다.
- 모바일에서는 입력 폼과 기간 필터가 세로로 정렬되어야 합니다.

### 5.3 공통 컴포넌트

| 컴포넌트 | 용도 |
| --- | --- |
| Header | 로그인 사용자, 로그아웃, 관리자 메뉴 표시 |
| Sidebar 또는 Top nav | 근무시간 입력, 조회, 감사 로그, 설정 이동 |
| Form | 회원가입, 로그인, 근무시간 입력, 기간 필터 |
| Table | 근무 기록, 변경 이력, 접속 로그, 웹훅 로그 조회 |
| Badge | 근무 기록 상태, 웹훅 처리 결과, 권한 표시 |
| Alert | 저장 성공, 검증 오류, 접근 거부, 웹훅 실패 표시 |
| Modal | 수정 확인, 삭제 또는 비활성화 확인 |
| Pagination | 장기간 로그와 근무 기록 목록 분할 |

## 6. 성공 기준

- 사용자는 회원가입과 로그인 후 본인 근무 기록을 등록할 수 있습니다.
- 사용자는 기간을 지정해 본인 근무 기록과 합계를 조회할 수 있습니다.
- 관리자는 사용자별 근무 기록과 변경 이력을 조회할 수 있습니다.
- 근무 기록 생성, 수정, 삭제, 로그인, 웹훅 처리 내역이 감사 로그에 남습니다.
- 허용되지 않은 IP 또는 잘못된 secret의 웹훅 요청은 거부되고 로그에 남습니다.
- 정상 웹훅 요청은 근무기간 요약을 텔레그램으로 전송합니다.

### 6.1 현재 검증 상태

- 완료: 회원가입, 로그인, 로그아웃, 내 정보 조회
- 완료: 본인 근무 기록 생성, 기간 조회, 요약, 수정, soft delete
- 완료: 일반 사용자 본인 기록 접근 제한
- 완료: 인증, 근무 기록 변경, 웹훅 처리 감사 로그 기록
- 완료: 웹훅 IP allowlist, shared secret, 중복 `requestId` 거부
- 부분 완료: 텔레그램 발송 로직은 구현되어 있으나, 운영 토큰과 chat id가 없는 로컬 환경에서는 발송 실패/스킵 상태로 검증했습니다.
- 미완료: 관리자용 화면과 감사 로그 조회 화면
- 미완료: 운영 rate limit과 백업 자동화

## 7. 로드맵

### 1단계: 프론트엔드 정리

- 기존 AI 파서 UI를 근무시간 입력 화면으로 정리
- 파싱 결과를 저장 전 확인 화면으로 분리
- Bootstrap 5 기반 공통 레이아웃 적용

### 2단계: 백엔드와 데이터베이스

- `index.php` 기반 라우팅 구조 추가
- `.env` 로더와 PDO DB 연결 클래스 구현
- 회원가입, 로그인 API 구현
- 사용자별 근무 기록 CRUD 구현
- MySQL 8.0 기준 데이터베이스 마이그레이션 작성

### 3단계: 감사 로그와 권한 관리

- 모든 주요 API에 인증 미들웨어 적용
- 근무 기록 생성, 수정, 삭제 시 감사 로그 자동 기록
- 관리자 조회 기능 추가
- 변경 이력 화면 추가

### 4단계: 웹훅과 텔레그램 연동

- IP allowlist
- shared secret 검증
- 중복 요청 방지
- 텔레그램 Bot API 연동
- 웹훅 처리 결과 로그 저장

### 5단계: 운영 보안 강화

- HTTPS 강제
- secure cookie
- CSRF 보호
- rate limit
- 백업 정책
- 로그 보존 정책
- 배포 환경변수 관리
